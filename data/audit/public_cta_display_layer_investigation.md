# Public CTA Display Layer Investigation

## Scope

This is an investigation note only. It does not change public pages, offer generation, booking URLs, seed selection, or scheduling rules.

Goal: add a public display layer for schedule offerings without exposing internal scheduling terms or flooding public pages with every generated seed.

Proposed public CTA tiers:

| Public CTA | Public meaning | Internal source |
| --- | --- | --- |
| Join this class | A real class already exists or has seated proof. | Enrollware class/session proof, roster/Class Report, hot-sync, registration signal, or equivalent confirmation. |
| Book this date/time | A preferred public-sellable opportunity near an existing or intended class stack. | Seed strategy output, stack/adjacent preference, selected public seed. |
| Check this time | Open instructor/location availability exists, but it should be collapsed into a cleaner availability block. | Public-sellable open availability offers after JOIN and BOOK are ruled out. |

Public pages must not expose internal terms such as hot_sync, seated, seed, stack seed, stack, adjacent, scheduler consumption window, source window, internal priority, audit/runtime/private fields.

## 1. Where Seated / Hot-Sync Proof Currently Lives

Likely sources:

- `data/sessions_current.json`
- `docs/data/schedule_future.json`
- `data/runtime/enrollware_sync/**`
- Class Report import path used by:
  - `scripts/build_sessions_current.py`
  - `scripts/build_schedule_future.py`
  - `scripts/enrollware_sync.py`
- Registration event bridge outputs:
  - `data/audit/appointment_seed_registration_matches.json`
  - `data/audit/enrollware_registration_events_normalized.json`

Current public class pages and schedule indexes primarily render from `docs/data/schedule_future.json`. That is the safest current source for “real class” rows because it is already public schedule data derived from the Class Report / Enrollware schedule pipeline.

Open question: the repo has multiple proof signals, but there is not yet one normalized field such as `seated_class_proof` or `real_class_status` that public rendering can consume directly.

## 4. Do Public Offer Records Already Contain Enough Fields For JOIN / BOOK / CHECK?

Partially.

Existing dynamic offers already include useful fields:

- `offer_id`
- `course_id`
- `course_title`
- `course_family`
- `date`
- `start_time`
- `appointment_display_start`
- `appointment_display_end`
- `scheduler_consumption_start`
- `scheduler_consumption_end`
- `scheduler_consumption_minutes`
- `setup_buffer_minutes`
- `cleanup_buffer_minutes`
- `instructor_person_id`
- `instructor_display_name`
- `location`
- `resource`
- `source_availability_window`
- `confidence`
- `reasons`

These fields are enough to classify normal open availability into CHECK blocks if public-sellable offers are already generated and filtered.

They are not enough by themselves to assign JOIN because JOIN requires a proof signal from the real class/session pipeline, not just dynamic availability.

They are mostly enough to assign BOOK when the offer appears in selected seed output and has valid appointment URL preview, but the repo does not yet have a normalized public display record that says “this seed is public-intended and should show as Book this date/time.”

## 5. Do Generated Open Seeds Share An Availability Block / Window Identifier?

Yes, mostly.

Dynamic offers carry `source_availability_window`.

For live calendar-derived windows this appears to come from `source_event_id` or a fallback such as `live_availability_snapshot[index]`.

For legacy windows this comes from local availability data.

This is a suitable first grouping key for CHECK blocks when combined with:

- course being viewed
- date
- location/resource
- instructor
- availability start/end

## 6. If No Block ID Exists, Can Grouping Be Safely Inferred?

Yes, but only as a fallback.

Safe inferred grouping key:

- course id or course family being viewed
- date
- instructor/person
- location/resource or matched appointment container location
- availability window start/end

Do not group only by date/location. That can incorrectly combine different instructors, resources, courses, or windows.

## 7. Course-Aware CHECK Block Clarification

The collapsed CHECK availability block must be course-aware.

For each course being viewed, the start-time selector may only show start times where the full scheduler consumption window fits inside the instructor/location availability block.

Use the true consumed time, not just the customer-facing class display time.

The calculation must include:

- course duration
- setup buffer
- cleanup buffer
- any scheduler consumption-window rules already used by the dynamic offer system

Example:

If instructor/location availability is 6:00 PM to 10:00 PM:

- A 45-minute skills session with required buffers may be selectable much later into the block.
- A 3-hour BLS Initial class may only show earlier start times.
- A course must never be selectable if its true consumed window would extend past the availability block end.

The CHECK block may display the overall availability window publicly, for example:

> Evening availability  
> 6:00 PM to 10:00 PM

But the opened selector must only show valid start times for the specific course being viewed.

Do not use the collapsed block as permission to offer every course until the block end. The course-specific valid-start calculation still controls the selectable times.

Implementation implication:

CHECK should group already-valid public-sellable offer records. It should not independently generate start times from the raw availability window unless it reuses the exact same course duration, setup buffer, cleanup buffer, conflict, container, and scheduler consumption checks used by `scripts/generate_dynamic_offers.py`.

## 10. Risks And Missing Data

Risks:

- JOIN can be misused if a dynamic seed is treated as a real class without Class Report / roster / registration proof.
- BOOK can be overused if every selected seed is shown publicly without operator policy.
- CHECK can become misleading if collapsed availability windows offer invalid start times for longer courses.
- CHECK grouping can accidentally expose too many theoretical possibilities if it groups raw availability instead of already-filtered offers.
- Public pages must not expose internal fields such as source offer ids, seed ids, scheduler consumption windows, or audit paths.

Missing or unclear:

- One canonical “real class proof” field for JOIN.
- One canonical public-intended flag for BOOK.
- Whether each public page should read a new public display JSON file or receive records during page generation.
- Whether CHECK selector URLs should be embedded directly in public HTML or resolved through a future recheck endpoint.

## Recommended Next Step

Create the read-only public display mapping script first.

It should classify existing records into JOIN / BOOK / CHECK without changing public pages.

The first version should:

1. Read real class rows from `docs/data/schedule_future.json`.
2. Read selected seeds from `data/audit/schedule_seeds_preview.json`.
3. Read URL previews from `data/audit/seed_appointment_url_preview.json`.
4. Read public-sellable offers from `data/audit/public_sellable_offers_preview.json`.
5. Build a report-only display preview.
6. For CHECK, group only already-valid offers by course-aware grouping keys.
7. Preserve course-specific valid start times and do not regenerate them from the collapsed availability block.

Only after this preview is reviewed should public pages render JOIN / BOOK / CHECK labels.
